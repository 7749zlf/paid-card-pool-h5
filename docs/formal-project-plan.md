# 短剧付费抽卡平台正式项目规划

## 1. 项目定位

本项目从当前 H5 交互雏形升级为正式前后端分离系统，核心业务是：短剧播放、付费节点、卡池抽卡、用户资产、订单支付、个人中心和后台运营管理。

当前静态 H5 只用于验证交互流程。正式项目中，支付、抽卡、用户资产、解锁状态、订单记录必须全部迁移到后端，前端只负责展示和发起请求。

## 2. 规模假设

目标用户量按 10 万量级准备，但需要区分不同指标：

- 10 万注册用户：常规单体后端 + MySQL + Redis 可以支撑。
- 10 万日活用户：需要 CDN、缓存、限流、队列、监控和可横向扩容后端。
- 10 万同时在线：视频带宽和 API 架构都要按高并发系统设计，必须使用视频 CDN/点播服务。

初期建议按“10 万注册用户、1-3 万日活、峰值 1000-5000 并发请求”设计，后续通过压测和监控逐步扩容。

## 3. 推荐技术架构

```text
用户
↓
CDN / WAF / DDoS 防护
↓
前端 H5 / 管理后台静态资源
↓
Nginx / API 网关
↓
后端 API 服务
↓
Redis：缓存、限流、幂等、分布式锁
↓
消息队列：支付回调、发奖、日志、通知
↓
MySQL：核心业务数据
↓
对象存储 + CDN / 视频点播
```

## 4. 技术选型建议

### 前端 H5

- Vue 3 + TypeScript + Vite，或 React + TypeScript + Vite。
- 播放页、付费节点、抽卡动画、个人中心继续沿用当前交互方向。
- 静态资源部署到 CDN 或静态托管，不直接放在后端服务器。
- 视频只预加载当前视频和下一条，避免流量浪费。

### 后台管理

- Vue 3 + TypeScript + Element Plus / Ant Design Vue。
- 独立后台项目，和用户 H5 分开部署。
- 后台功能包括视频管理、卡池配置、订单查询、用户资产、抽卡记录、运营配置、管理员权限和审计日志。

### 后端

推荐优先级：

1. Java Spring Boot：适合支付、订单、后台、事务一致性和长期维护。
2. Go：并发性能好，部署轻，但后台生态和业务开发需要更多工程约束。
3. Node.js / NestJS：开发快，适合小团队，但高并发和支付资产类业务要更严格控制事务和类型。

建议初期使用 Spring Boot 模块化单体，不建议一开始直接微服务。等用户量和团队规模上来后，再拆分订单、支付、抽卡、视频、用户等服务。

### 数据和中间件

- MySQL 8：核心业务数据库。
- Redis：缓存、限流、分布式锁、验证码、幂等标记。
- RabbitMQ：初期用于支付回调、发奖、异步日志、通知。
- 对象存储：视频封面、卡牌图片、静态上传资源。
- 视频点播/CDN：短剧视频播放，不让业务服务器直接输出视频流。
- Docker + Nginx：部署和反向代理。

## 7. 关键业务流程

### 视频付费节点流程

```text
用户播放视频
↓
到达付费节点
↓
前端展示付费提示
↓
用户点击支付
↓
后端创建订单
↓
支付渠道完成支付
↓
后端收到支付回调并验签
↓
写入用户解锁状态
↓
前端查询并继续展示内容
```

### 单次抽卡流程

```text
用户点击抽取一次
↓
后端创建抽卡支付订单
↓
用户支付
↓
支付回调验签
↓
后端执行抽卡事务
↓
写入 draw_record 和 user_card
↓
前端展示抽卡动画
↓
前端展示抽到的卡牌详情
```

### 支付回调幂等要求

支付回调可能重复发送，后端必须保证同一支付流水只处理一次。

处理方式：

- 订单表有唯一订单号。
- 支付流水表有唯一支付流水号。
- 回调处理先查订单状态。
- 已支付订单直接返回成功。
- 未支付订单在事务内更新状态和发放权益。
- Redis 可加短期幂等锁，数据库唯一索引是最终保障。

## 8. 初版数据库表设计

以下是第一版核心表，后续开发时再细化字段类型和索引。

### 用户相关

- `users`：用户主表。
- `user_auths`：登录方式表，支持手机号、邮箱、第三方登录。
- `user_profiles`：用户资料。

### 视频相关

- `dramas`：短剧主表。
- `episodes`：剧集表。
- `video_assets`：视频资源表。
- `watch_histories`：观看历史。
- `user_favorites`：收藏。
- `user_likes`：点赞。

### 支付订单相关

- `orders`：业务订单。
- `payment_transactions`：支付流水。
- `payment_callbacks`：支付回调原始记录。
- `refund_orders`：退款订单，预留。

### 卡池抽卡相关

- `card_pools`：卡池。
- `cards`：卡牌。
- `card_pool_cards`：卡池卡牌关联和概率配置。
- `draw_records`：抽卡记录。
- `user_cards`：用户已拥有卡牌。
- `user_unlocks`：用户解锁状态。

### 后台和审计

- `admin_users`：管理员。
- `roles`：角色。
- `permissions`：权限。
- `admin_operation_logs`：后台操作日志。
- `system_configs`：系统配置。

## 10. 高并发设计要点

### 静态资源

- 前端 H5、后台管理、图片资源全部走 CDN。
- JS/CSS 使用版本化文件名和长缓存。
- 首页 HTML 使用较短缓存，避免发布后无法更新。

### 视频播放

- 视频必须使用对象存储 + CDN 或视频点播。
- 不允许业务 API 服务直接输出视频。
- 启用防盗链、Referer 校验或签名 URL。
- 控制预加载策略，只加载当前和下一条。

### 后端 API

- 服务无状态，支持横向扩容。
- 热点数据进入 Redis。
- 读多写少配置缓存。
- 写操作必须幂等。
- 抽卡和支付使用事务 + 唯一索引保障一致性。

### 数据库

- 所有高频查询提前设计索引。
- 分页接口避免深分页。
- 订单、抽卡记录按时间和用户维度建索引。
- 上线后开启慢 SQL 监控。
- 用户规模继续增长后再考虑读写分离和分库分表。

### Redis

- 登录态或会话缓存。
- 验证码。
- 接口限流。
- 支付回调短期幂等锁。
- 热点配置缓存。

### 队列

- 支付回调后异步通知。
- 发奖补偿任务。
- 操作日志异步落库。
- 数据统计异步处理。

## 11. 安全要求

### 必须实现

- HTTPS。
- WAF / CDN 基础防护。
- Nginx 限流。
- 登录接口限流。
- JWT 或 Session 鉴权。
- 后台管理 RBAC 权限。
- 管理后台二次验证，至少预留。
- SQL 注入防护。
- XSS 防护。
- CSRF 防护，按登录态方案决定。
- 支付回调验签。
- 抽卡和支付幂等。
- 用户越权校验。
- 敏感数据脱敏。
- 管理员操作审计。

### 高风险点

- 前端伪造支付成功。
- 重复请求抽卡。
- 支付回调重复发奖。
- 用户越权查看他人卡包。
- 管理后台权限过大。
- 视频资源被盗链刷流量。
- 日志泄露手机号、邮箱、支付参数。
- 数据库没有可恢复备份。

## 14. 备份和恢复

- MySQL 每天全量备份。
- 核心业务表增加更高频备份或 binlog 保留。
- 备份文件异地保存。
- 定期做恢复演练。
- 对象存储开启版本控制或误删保护。
- 后台关键操作进入审计日志。

备份如果没有恢复演练，等同于不可验证。

## 17. 当前原型迁移策略

当前 H5 原型中的功能可以按以下方式迁移：

- `localStorage` 解锁状态迁移为后端 `user_unlocks`。
- 前端随机抽卡迁移为后端抽卡服务。
- 前端模拟订单迁移为后端订单表和支付流水表。
- 静态视频地址迁移为视频点播或 CDN URL。
- 个人中心数据改为接口查询。
- 后台管理新增视频、卡池、订单、用户资产配置能力。
